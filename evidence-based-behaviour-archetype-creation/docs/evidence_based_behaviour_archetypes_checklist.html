Checklist & Method Guide

1
Decision & data readiness
0/6
✓
CriticalConfirm you have the minimum viable evidence base
≥10 behavioral records OR ≥3 interviews. Below these thresholds, produce a pattern summary only — not a named persona profile.
✓
MixedWrite the research question in one sentence
If you cannot, the scope is too broad. Vague questions produce misleading segments.
✓
MixedDefine the population boundary explicitly
Who is in scope, who is not. Ambiguous populations produce misleading segments.
✓
CriticalConfirm personas are the right artefact for this question
Consider: JTBD for task flows, journey maps for emotional experience, segmentation reports for sizing. Personas need a team that will actively use them.
✓
GenAIIf using AI generation: declare the accessibility-quality paradox to stakeholders
Amin et al. (2025): GenAI lowers creation barriers but evaluation still requires expertise. 45% of reviewed articles had no evaluation framework. Ease of generation does not reduce the evaluation obligation.
✓
MixedConduct inclusivity pre-check before data collection
Which groups are over-represented? Which are absent? Is community validation planned for marginalised populations?
2
Research planning & source selection
0/5
✓
MixedSelect source types appropriate to the research question
Analytics → behavioural patterns only. Survey → behavioural + optional demographics. Interview → motivations, goals, quotes. Do not cross capabilities.
✓
QuantVerify sample sizes meet directional confidence thresholds
Analytics/survey: ≥30 records. Interviews: 8–12 sessions for thematic saturation. Below these, label outputs as hypothesis-generating only.
✓
QuantMap data collection instruments to generator field schema
usage_frequency, features_used, primary_device, usage_context, pain_points — always. Optional fields (age, tech_confidence) in survey only.
✓
GenAIIf using synthetic data: label it explicitly and plan real-user validation
Amin et al. (2025): 32% of GenAI persona articles used synthetic data. LLMs generate US-centric, elite-skewed personas from synthetic input without geographic constraints. Synthetic data ≠ real user research.
✓
MixedDeclare the minimum signal threshold for this study
Default is 3 occurrences. Raise to 5+ for high-stakes design decisions. Document this decision before analysis begins.
3
Quantitative segmentation & analysis
0/5
✓
QuantRun behavioral pattern analysis across all behavioral sources
Usage frequency, device, context, features, pain points. Read raw distributions before the summary — the summary is derived from them.
✓
QuantValidate cluster stability and internal coherence
For algorithmic segmentation: silhouette scores, Davies–Bouldin index, or within-cluster sum of squares. Document algorithm and hyperparameters.
✓
QuantSurface optional fields only when present in source data
Age, tech_confidence, location_type — show as distributions with source label. Never infer from proxy signals. Declare absent fields explicitly.
✓
GenAICheck AI-generated demographic outputs for LLM default bias
Amin et al. (2025): LLMs skew US-centric, elite-educated, and majority demographic. Compare generated distributions to real user data (PG4). Document deviations.
✓
MixedGenerate example segments as emergent, overlapping, illustrative slices
Not persona types. The most common observed combinations of frequency, device, and context. A user may fit multiple segments over time.
4
Qualitative enrichment
0/5
✓
QualConduct user interviews to enrich quantitative segments
Salminen et al. (2021) Gap E: purely data-driven approaches risk losing in-depth user insights. Interviews ground the numbers in real human context.
✓
QualAdd only explicitly stated goals, needs, and motivations
Every qualitative claim traces to an interview quote or stated response. "Inferred from behaviour" is not a valid source for motivation claims.
GenAIIf LLM used for thematic analysis: document it as an AIEnrichmentRecord
Record stage, model, version, prompt reference, and whether outputs were validated against source transcripts (PG3). Unvalidated AI thematic outputs are hallucination candidates.
✓
CriticalVerify all qualitative claims are grounded in real data — not LLM inference
Muller & Seaborn (2025): Potemkin personas appear real but are staged computational artifacts. Every attribute must trace to a specific source record.
✓
QualInclude only real interview quotes — never fabricated defaults
If no interviews were conducted, the quotes field is empty. An LLM-generated "representative quote" is a fabrication.
5
Profile craft
0/6
✓
MixedUse a descriptive behavioural label, not a fictional archetype name
"Daily desktop user — work context, broad feature use" not "Alex the Power User". Archetype names assign identity before examining data.
✓
QuantShow optional demographic fields as distributions with source label
"Age: 26–38, mean 31 — Q3 survey, n=28". Never assign a single age or tech confidence value to a persona profile.
✓
CriticalEnsure photos and illustrative names reflect the actual demographic distribution
Default AI-generated images skew majority demographics. This is a documented bias pattern. Challenge any default choice.
✓
MixedWrite scenarios grounded in observed data — not invented context
Context from usage_context distribution. Goal from needs_and_goals block. Friction from frustrations block. If data doesn't support a scenario, don't write one.
✓
GenAIInclude AI involvement in the profile footer
Which stages used LLM assistance, which models, and which prompt version (PG3). This is required for reproducibility and stakeholder trust (Amin et al., 2025).
✓
CriticalInclude a condensed limitations block in every profile footer
Data sources with record counts, confidence note, top 2–3 limitations, and date. This is non-negotiable — it distinguishes an evidence-based profile from a Potemkin persona.
6
Governance & maintenance
0/5
✓
MixedAssign an owner responsible for updates
Every output must have a named owner. Without one, persona maintenance does not happen and stale outputs accumulate as liabilities.
✓
MixedDocument the intended use — which decisions this output informs
Personas made for one purpose get misapplied to others. Stating intended use limits scope creep and prevents downstream misuse.
✓
MixedRecord the date of the most recent evidence source
Jung et al. (2019): significant persona drift occurs over two years. Evidence older than 12 months with a changed product should be treated as expired until re-validated.
✓
MixedDefine the review triggers for this output
Review when: product changes materially, new behaviours emerge in data, evidence contradicts the model, or team is making a new class of decision not covered by the original brief.
✓
CriticalDo not treat the output as static doctrine
A persona that is never updated is not a research asset — it is a liability. Commit to a review cadence before publishing, not after the first time someone questions it.
7
Evaluation (most under-resourced stage)
0/8
✓
EvalEvaluate persona description quality using the Persona Perception Scale (PPS)
Salminen et al. (2020). Five dimensions: consistency, clarity, completeness, credibility, fluency. Validated instrument for profile quality assessment.
✓
EvalEvaluate persona behaviour: does it act consistently with real user patterns?
Amin et al. (2025): behaviour evaluation asks whether personas act like real humans — distinct from whether they look credible. Test decision-making accuracy against real user data.
✓
EvalEvaluate outcomes and impact: do team decisions improve when using the persona?
Task-based evaluation: give team members a design question, compare decisions made with vs. without the persona. Measure usability with SUS if deploying a persona system.
✓
GenAIPerform hallucination check: trace every factual claim to a source record
Amin et al. (2025): only 11.5% of reviewed articles address hallucination. Claims without a traceable source record are hallucination candidates. Remove or label as speculative.
✓
GenAIPerform consistency testing: run the generator 3× and compare outputs (PG6)
Attributes that shift substantially across runs are unstable and likely partially hallucinated. Unstable outputs should not be used in design decisions without further validation.
✓
GenAIAvoid circularity: use a different model (or human evaluators) for evaluation
Amin et al. (2025): using the same LLM to generate and evaluate personas tests model-specific patterns, not genuine quality. This is a documented validity threat.
✓
GenAIValidate against demographic benchmarks or real user data (PG4)
Compare age, geographic, and occupational distributions in generated personas against your actual user base. Document deviations. LLM defaults overrepresent elite universities and US geographies.
✓
CriticalApply PG7 human oversight: domain expert review before design use
Non-negotiable. For personas representing marginalised or vulnerable populations, community representatives must be involved. No fully automated pipeline should make design decisions without human review.
Salminen et al. (2021) identified three historical periods in data-driven persona development. Amin et al. (2025) documents a fourth: Augmentation (2022–present), where GenAI augments rather than replaces human involvement. Click any card to expand.
📊
K-means clustering
Period 1 — Quantification (2005–2008)
Partitions users into k clusters based on behavioural or demographic variables. Most dominant method 2009–2014.
PROSimple, interpretable, scales to large datasets
CONRequires pre-specifying k; sensitive to outliers; assumes spherical clusters
🔤
Latent Semantic Analysis (LSA)
Period 1 — Quantification
Derives segments from text corpora (support tickets, reviews) by identifying latent topic structures.
PROExcellent for text-rich datasets; uncovers latent themes
CONCannot incorporate behavioural metrics; requires large text corpus
🧮
Non-negative Matrix Factorisation (NMF)
Periods 2–3
Factorises the user-behaviour matrix into non-negative components, each a user archetype. Basis of the APG system.
PROHandles mixed behavioural + demographic data; soft memberships; gold-standard APG
CONLess interpretable than k-means; requires non-negative matrices
📋
Survey-based (Survey2Persona)
Period 3 — Digitalization
Generates personas from survey data. PersonaCraft (Jung et al., 2025) integrates LLMs into this pipeline with empirical evaluation.
PROAccessible; surveys provide rich qualitative context; no analytics access needed
CONSurvey responses may be biased; smaller samples than analytics
🤖
LLM-assisted generation
Period 4 — Augmentation (2022–present)
LLMs generate persona narratives from structured data, interview transcripts, or synthetic inputs. 86% of GenAI persona research uses GPT models.
PROCompresses timelines; enables narrative richness; interactive/conversational personas
CONHallucination risk; single-vendor dependency; 45% of research lacks evaluation; US-centric bias
👥
Manual / interview-based (enrichment)
All periods — qualitative layer
Contextual enquiry, in-depth interviews, ethnographic observation. Used to enrich quantitative or AI-generated segments with human insight.
PROCaptures nuance, motivation, and emotion that algorithms miss; builds empathy
CONNot scalable; researcher bias possible; cannot be replaced by LLM inference
Four GenAI innovation types identified by Amin et al. (2025): input source innovations (multimodal, unstructured data), process innovations (direct attribute specification, segmentation-free approaches, automated pipelines), prompt engineering innovations (two-stage prompting, few-shot learning, chain-of-thought), and output format innovations (narrative-driven, multimodal, interactive/conversational personas). Each carries distinct evaluation requirements.
Seven practical guidelines from Amin et al. (2025) Table 8, plus three additional risks identified in the review. These apply whenever LLMs are used at any stage of persona development.
✓ The Good
  • Compresses timelines from weeks to hours
  • Enables conversational/interactive personas
  • Improves resource sharing and reproducibility
  • Enables iterative human-AI co-creation
  • New agentic personas that simulate behaviour
  • Multimodal representation formats
⚠ The Bad
  • 45% of articles lack evaluation frameworks
  • 86% use only GPT — single-vendor dependency
  • Only 11.5% address hallucination detection
  • Evaluation practices are ad hoc and fragmented
  • Accessibility-quality paradox: easy to create, hard to validate
✗ The Ugly
  • 42% of articles report no human-AI collaboration
  • Fully automated pipelines erase real users
  • Potemkin personas: staged, not grounded
  • Shifts control from communities to algorithms
  • Reduced semantic diversity vs. human-crafted personas
  • Synthetic data → AI generation → AI evaluation = closed loop
Guideline Requirement Implementation
PG1 Multi-model validation Use ≥2 different GenAI systems for validation. Different models exhibit distinct biases. If using Claude for generation, validate key outputs against GPT or Gemini, or against real user data.
PG2 Structured output formats Deploy JSON/CSV with explicit consistency criteria for programmatic validation. python persona_generator.py json
PG3 Prompt documentation Document prompts, parameters, model versions, and dates. Prompt sensitivity significantly affects outputs. Use AIEnrichmentRecord to capture stage, model, prompt_reference per source.
PG4 Demographic benchmarking Validate against real user data. LLM-generated personas systematically deviate from real populations. Compare optional_fields distributions to real user analytics. Flag US-centric or elite skew.
PG5 Acceptance thresholds Pre-register quality criteria before generating to prevent confirmation bias. Set min_signal_count before running. Define PPS score threshold before evaluating.
PG6 Consistency testing Run generator 3× on same data. High variability = unstable representations. Attributes that shift across runs are hallucination candidates. Do not use in design decisions.
PG7 Human oversight Domain expert or community review before design use. Automated methods miss cultural details. Non-negotiable for marginalised populations. At least one human checkpoint per phase.
Anti-circularity Do not use the same LLM to generate and evaluate. Tests model patterns, not genuine quality. Use a different model or human evaluators for evaluation phase.
Anti-Potemkin Every attribute must trace to a source record. Staged personas appear real but aren't grounded. Muller & Seaborn (2025): apply source-tracing check before profile release.
Hallucination check Trace every factual claim to a source record before use. Flag unvalidated AI-enriched stages in ai_involvement output block.
Five gaps from Salminen et al. (2021) remain active — Amin et al. (2025) confirms most persist in the GenAI era, with some showing early progress (resource sharing improved) and others worsening (evaluation gaps deepened).
A
Shared resources and provenance
Most data-driven persona research shared no datasets, code, or prompts. GenAI research shows improvement — 61% share at least one resource type — but 38% still share nothing. The DataSource registry and AIEnrichmentRecord in persona_generator.py address this directly.
B
Evaluation methods
This is the deepest ongoing gap. 45% of GenAI persona articles (Amin et al., 2025) have no evaluation framework. Evaluation must cover four targets: description quality, behavioural consistency, generation process, and outcomes/impact. Using PPS alone is insufficient.
C
Standardisation
No agreed lifecycle, terminology, or reporting standard exists. GenAI adds new variation: LLM family, version, temperature, prompt strategy. The output schema in persona_generator.py is the reference implementation for this skill.
D
Inclusivity and algorithmic fairness
GenAI amplifies this gap. Amin et al. (2025) documents: algorithmic othering, elite institution overrepresentation, gender and age bias, platform-specific sampling bias, and AI exacerbation of existing stereotypes. 80% of personas in one study showed stereotypical bias. Community validation is required for marginalised populations.
E
Risk of losing in-depth user insights
The "Ugly" risk identified by Amin et al.: fully automated pipelines erase real users. 42% of GenAI persona articles have no human-AI collaboration model. Qualitative enrichment and human oversight remain non-negotiable. Interview evidence must be preserved separately from behavioural aggregates.
Additional anti-patterns from practice (File 2 methodology): beyond the five Salminen gaps, practitioner methodology identifies three further failure modes: the Archetype Trap (forcing users into power/casual/business/mobile-first before examining data), Confidence Theater (a formula creating the appearance of rigour without real rigour — replace numeric scoring with transparent confidence notes), and the Output-First Mistake (optimising for the artifact rather than the decision it should support — state the decision upfront and remove components that don't serve it).
Three-part pre-publication checklist (File 2):

Research quality: Research question is clear · ≥2 evidence sources used · Major claims are traceable · Contradictions documented · Weak signals labelled as tentative

Synthesis quality: Patterns described before categories proposed · Segments are clearly warranted · Example segments are overlapping and illustrative · No fabricated names, photos, or quotes · Psychographic claims directly evidenced or omitted · Output distinguishes observation from interpretation

Decision quality: Output helps the team make a real decision · Design implications are evidence-linked with "because" fields · Limitations are stated clearly · Confidence expressed as a note, not a formula · Output includes owner, intended use, and review trigger

New — essential GenAI framework
Creating and Evaluating Personas Using Generative AI: A Scoping Review of 81 Articles
Amin, Salminen, Ahmed, Tervola, Sethi & Jansen — arXiv:2504.04927v2, 2025
The primary update to this skill. Systematic review of 81 GenAI persona articles 2022–2025. Source of the Good/Bad/Ugly framework, seven practical guidelines (PG1–PG7), and the accessibility-quality paradox. Read alongside Salminen et al. (2021).
Essential — foundational survey
A Survey of 15 Years of Data-Driven Persona Development
Salminen, Guan, Jung & Jansen — IJHCI, 2021
The systematic review underlying this skill's five research gaps. Reviews 77 data-driven persona articles 2005–2020. Still the foundational reference for method selection, evaluation standards, and the three historical periods of DDPD.
New — current state of the art
PersonaCraft: Leveraging Language Models for Data-Driven Persona Development
Jung, Salminen, Aldous & Jansen — IJHCS, 2025
The most current empirically evaluated LLM-assisted persona system. Integrates data collection, analysis, and generation into a unified pipeline with proper evaluation. Reference implementation for survey-based GenAI persona generation.
Essential — methodology
A Literature Review of Quantitative Persona Creation
Salminen, Guan, Jung, Chowdhury & Jansen — CHI, 2020
Direct precursor to the 2021 survey. Systematic categorisation of quantitative methods. Read first for method selection guidance.
Important — evaluation instrument
Persona Perception Scale
Salminen, Santos, Kwak, An, Jung & Jansen — IJHCS, 2020
Validated instrument for measuring persona profile quality across credibility, completeness, clarity, consistency, and fluency. Required for Phase 5 description evaluation.
New — critical risk paper
Stepford Twins and Potemkin Engineering: A Critique of Synthetic Personas
Muller & Seaborn — Synthetic Data Workshop, Aarhus, 2025
Defines the Potemkin persona concept — profiles staged to appear real but disconnected from actual users. Essential reading before any AI-assisted profile craft step.
New — human-AI workflow
Understanding Human-AI Workflows for Generating Personas
Shin, Hedderich, Rey, Lucero & Oulasvirta — DIS, 2024
Defines three collaboration models: LLM-auto (fully automated), LLM-grouping (human selects characteristics), LLM-summarizing (human clusters, LLM writes narratives). Provides a practical taxonomy for choosing a human-AI collaboration level.
Important — bias and fairness
Rethinking Personas for Fairness: Algorithmic Transparency and Accountability
Salminen, Jung, Chowdhury & Jansen — 2020
Introduces persona fairness as a concept. Proposes methods for auditing algorithmic bias. Directly addresses Gap D.
Important — hallucination in HCI
Evaluating LLMs in Generating Synthetic HCI Research Data: A Case Study
Hämäläinen, Tavast & Kunnari — CHI, 2023
Empirical evidence of hallucination risk in LLM-generated HCI research data. Directly relevant to Phase 3 hallucination detection and Phase 5 validation requirements.
Important — demographic bias evidence
Evaluation of LLM Biases Towards Elite Universities: A Persona-Based Exploration
Gupta & Ranjan — arXiv, 2024
Documents that LLM-generated personas overrepresent elite universities (72% in generated vs. 8.6% in actual data). Directly motivates PG4 demographic benchmarking.
Foundational
The Inmates Are Running the Asylum / About Face 3
Cooper, A. — 1999 / 2007
The origin of the persona methodology. Provides the conceptual and philosophical foundation that all subsequent data-driven and GenAI work builds upon or critiques.
Foundational
The Persona Lifecycle: Keeping People in Mind Throughout Product Design
Pruitt, J. & Adlin, T. — Morgan Kaufmann, 2006
The most comprehensive qualitative persona creation guide. Defines the lifecycle model that data-driven and AI-assisted research augments rather than replaces.
